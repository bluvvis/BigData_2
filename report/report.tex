\begin{abstract}
We implement an offline text indexer using Hadoop MapReduce, store the inverted index and BM25 statistics in Apache Cassandra, and rank queries with BM25. Document extraction from a Wikipedia parquet subset uses PySpark. The report explains design choices and demonstrates a successful run over 100 documents with command-line search.
\end{abstract}

\section{Methodology}

This section states \emph{what} each component does and \emph{why} it is shaped that way, so the implementation can be reproduced and graded against the assignment criteria.

\subsection{Execution environment}
The stack follows the course template: Docker Compose runs a two-node Hadoop cluster (\texttt{cluster-master}, \texttt{cluster-slave-1}) plus \texttt{cassandra-server}. The master container mounts \texttt{app/}, starts HDFS and YARN via \texttt{start-services.sh}, creates a Python virtual environment, and runs the end-to-end script \texttt{app.sh}. Cassandra is limited to a small heap (512\,MB) so the stack stays stable on typical laptops. This split matches the assignment: batch indexing on Hadoop, persistent index in Cassandra, and a Spark-based entry point for queries (\texttt{query.py} / \texttt{search.sh}).

\subsection{Data preparation (PySpark)}
\textbf{Requirement:} no Pandas or single-machine dataframe libraries for preparation; use PySpark.

\textbf{Input:} one parquet file \texttt{a.parquet} (Kaggle Wikipedia extract) with at least \texttt{id}, \texttt{title}, and \texttt{text}. Categories are ignored.

\textbf{Procedure:} \texttt{prepare\_data.py} reads the parquet with Spark, filters null text, samples rows, and keeps exactly $n=100$ documents (as required for the demonstration scope in our configuration). For each row it writes a plain UTF-8 file under \texttt{data/} named \texttt{<doc\_id>\_<title>.txt} with spaces in the title mapped to underscores and non-ASCII titles mapped to an ASCII slug so HDFS and shell tools handle paths reliably. In parallel it builds \texttt{indexer\_input.txt}: one line per document with tab-separated \texttt{doc\_id}, \texttt{title}, and full \texttt{text}. That single file is the streaming input for one MapReduce job, which is simpler than pointing Hadoop at hundreds of small files and matches a typical ``corpus file'' pattern.

\textbf{Why Spark in \texttt{local[1]} mode here:} the driver runs inside the master container; a local Spark driver avoids YARN acceptance delays and driver OOM from a full \texttt{count()} on the parquet. The preparation step still uses the Spark DataFrame API as required.

\textbf{HDFS layout:} \texttt{prepare\_data.sh} uploads every \texttt{data/*.txt} to \texttt{/data} (assignment-style document store) and uploads \texttt{indexer\_input.txt} to \texttt{/indexer/input/} for the indexer.

\subsection{Indexer (Hadoop MapReduce streaming)}
\textbf{Goal:} build a vocabulary, inverted index, and per-document statistics sufficient for BM25.

\textbf{Pipeline:} one MapReduce job (\texttt{mapper1.py}, \texttt{reducer1.py}) controlled by \texttt{create\_index.sh}. The mapper tokenizes text (lowercasing, alphanumeric tokens, minimum length), emits postings per term, and the reducer aggregates postings per term into a single inverted-index record per line. Output is written to HDFS under \texttt{/indexer/output} with one reducer task so the index is one partition, then copied to \texttt{/indexer/data/index} for loading.

\textbf{Design choice --- streaming and \texttt{-files}:} map tasks may run on workers without the \texttt{/app} bind mount. The script passes \texttt{-files} so \texttt{mapper1.py} and \texttt{reducer1.py} are shipped to all nodes; mapper and reducer are invoked by basename. This avoids ``cannot run program /app/mapreduce/mapper1.py'' on slaves.

\textbf{Trade-off:} a single reducer serializes the reduce phase but yields one output file, which satisfies the ``one partition'' style layout and keeps \texttt{store\_index.sh} simple.

\subsection{Storage in Cassandra}
\textbf{Keyspace:} \texttt{search\_engine}, replication factor 1 (single Cassandra node), \texttt{SimpleStrategy}.

\textbf{Tables (minimal design as required):}
\begin{itemize}[leftmargin=*]
  \item \texttt{inverted\_index} --- primary key \texttt{term}; columns \texttt{document\_frequency}, \texttt{num\_docs}, \texttt{postings\_data} (serialized postings from the reducer) for scoring.
  \item \texttt{documents(doc\_id, title, doc\_length)} --- supports result display and document length $|d|$ in BM25.
  \item \texttt{statistics(key, value)} --- holds corpus-level scalars, notably \texttt{num\_docs} $N$ and \texttt{avg\_doc\_length} for length normalization.
\end{itemize}

\textbf{Loader:} \texttt{app.py} creates the schema if missing, reads the HDFS index file, and bulk-inserts into Cassandra. Using the Python \texttt{cassandra-driver} keeps the loading script straightforward for a course-sized index; a Spark--Cassandra connector would scale better for very large indexes.

\subsection{Search and BM25 ranking}
\textbf{Interface:} \texttt{search.sh} accepts a query string and runs \texttt{query.py} via \texttt{spark-submit} (default \texttt{local[1]} in our scripts for reliability; YARN is optional via \texttt{USE\_YARN=1}).

\textbf{Retrieval:} for each query term, the client reads the row from \texttt{inverted\_index}, parses postings, and accumulates candidate documents. Document titles and lengths come from \texttt{documents}; $N$ and average length from \texttt{statistics}.

\textbf{Scoring:} we use BM25 with $k_1=1.5$ and $b=0.75$. IDF uses the common lower-bounded form:
\begin{equation}
\mathrm{IDF}(t) = \ln\frac{N - \mathrm{df}(t) + 0.5}{\mathrm{df}(t) + 0.5}.
\end{equation}
Per-term score:
\begin{equation}
\mathrm{BM25}(t,d) = \mathrm{IDF}(t)\cdot
\frac{f(t,d)\,(k_1+1)}{f(t,d) + k_1\left(1 - b + b\,\dfrac{|d|}{\mathrm{avgdl}}\right)}.
\end{equation}
Scores add over query terms; the top 10 $(\mathrm{doc\_id}, \mathrm{title})$ pairs are printed.

\textbf{Spark in \texttt{query.py}:} ranking logic uses Cassandra for data access; a minimal SparkContext and a trivial RDD operation satisfy the ``PySpark application'' requirement while keeping latency low on a small index. The same script can be submitted to YARN when the cluster archive is configured.

\section{Demonstration}

\subsection{How to run the repository}
\begin{enumerate}[leftmargin=*]
  \item Install Docker and Docker Compose.
  \item Download \texttt{a.parquet} from the course dataset link (Kaggle Wikipedia parquet) and place it in the \texttt{app/} directory next to the scripts.
  \item From the repository root (where \texttt{docker-compose.yml} lives), run:
  \begin{verbatim}
docker compose up
  \end{verbatim}
  The \texttt{cluster-master} container starts SSH and Hadoop services, installs Python dependencies, runs \texttt{prepare\_data.sh} (100 documents), \texttt{index.sh} (MapReduce + Cassandra load), then runs several example searches via \texttt{search.sh}.
  \item Optional: copy \texttt{.env.example} to \texttt{.env} and set \texttt{STAY\_ALIVE\_AFTER\_PIPELINE=1} if you need the master container to stay alive after the pipeline for extra manual queries (\texttt{docker exec -it cluster-master bash}, then \texttt{cd /app \&\& bash search.sh "your query"}).
\end{enumerate}

\subsection{Explanation of retrieved results}
For the query illustrated in Figure~3, documents that appear high in the ranking typically (i)~contain multiple query terms or repeated important terms, raising $f(t,d)$ in the numerator of BM25 while the saturation denominator prevents a single term from dominating extremely long texts; (ii)~have length $|d|$ near the corpus average, so the length penalty $(1-b)+b|d|/\mathrm{avgdl}$ is moderate; (iii)~benefit from higher IDF on less frequent query terms, which down-weights stopword-like terms shared by many documents. Documents with no overlap on any query term do not enter the accumulator and are absent from the top ten.

The script prints debug lines on stderr (corpus $N$, average length, tokenized query), which helps verify that statistics in Cassandra match expectations after load.

\subsection{Reflection}
\begin{itemize}[leftmargin=*]
  \item \textbf{End-to-end complexity:} coordinating HDFS paths, streaming job packaging, and Cassandra schema in Docker mirrors a small production batch pipeline; the main fragility is environment-specific (memory, YARN scheduling), which we mitigate with local Spark for preparation and default local mode for search unless YARN is explicitly enabled.
  \item \textbf{Quality vs.\ scale:} with only 100 documents, IDF is noisy and many queries return a thin tail of weakly relevant hits; BM25 is still the right baseline because it combines term frequency, document frequency, and length in one principled score.
  \item \textbf{Possible extensions:} multiple MapReduce jobs for separate vocabulary / stats files, or Spark--Cassandra bulk load for larger corpora, without changing the BM25 definition.
\end{itemize}
